## 1. 目的・ユースケース

保存済み構成（builds）を **2つ選んで対戦させた場合**、双方の機体について
**「各武装で相手にどれだけのダメージを出せるか」** を計算・一覧表示する。

- 入力: 構成A・構成B（既存の保存構成。`App.savedBuilds` の1要素）
- 出力（双方向）:
  - A の各武装 → B に与えるダメージ（B の装甲・耐性・三すくみ・特殊補正を考慮）
  - B の各武装 → A に与えるダメージ
- 「[[基準構成との比較]]」が *ステータス* の比較なのに対し、本機能は *実戦ダメージ* の比較。
  「どっちの構成がこの相手に強いか」を武装単位で見られるようにするのが狙い。

既存の「火力期待値（総合火力スコア）」は **武器を持たない抽象的な与ダメ倍率**でしかない。
本機能は **実際の武器威力 × 補正** を用いた具体ダメージへ踏み込む点が新しい。

## 2. 現状とのギャップ

> ⚠ **武器（武装）データは現在 `ms_data.json` に収録されていない。**
> ただし **Wiki の機体ページには武装表として詳細に載っており**（2026-06-09 実機確認、§2-2）、
> *データが無い* のではなく *まだパースしていない* だけ。→ 障壁は「武装パーサ新設」であり、
> データ取得自体は既存の curl_cffi パイプラインで可能。当初想定よりリスクは低い。

`data/ms_data.json`（574機体）の各機体が現在持つのは以下のみ:

```
top: name, pageId, category, ground, space, envGround, envSpace,
     levels, slots, skills, enhancements, wikiUrl
levels[lv]: cost, hp, ballistic_armor, beam_armor, melee_armor,
            shooting_correction, melee_correction, speed, boost_speed,
            thruster, turn_speed_ground, turn_speed_space
```

→ **武器の威力・属性・よろけ値・特殊効果は未取得**。本機能はまず
**武装表パーサ新設＋ms_data.json への weapons 追加**が前提条件になる（§5・§8-Phase1）。

### 2-2. Wiki 武装表の実構造（リゼルN型 pageId 7554 で確認）

機体ページに武器ごとのテーブルが複数あり（リゼルN型は射撃/格闘あわせて7表）、
**武器種別でカラム構成が異なる**。確認できた実カラム:

```
射撃(ビーム/実弾):  武器名 | LV | 威力 | OHまでの弾数 or 弾数 | 発射間隔 | OH復帰時間/リロード時間 | 切替時間 | 射程 | (DPS) | 備考 | 必要DP
格闘:               武器名 | LV | 威力 | クールタイム | 武装切替 | 備考 | 必要DP
```

**リゼルN型の武装は全8種**（当初「BR と サーベルだけ」と誤報告 → 訂正）:
1. `リゼルN型用ビーム・ライフル`（ビーム射撃）威力 **2200/2400**、射程300m、備考「集束可/集束時大よろけ/倍率1.727/よろけ値30%」
2. `リゼルN型用B・サーベル`（格闘）威力 **2500/2700**、クールタイム2秒、「ヘビーアタック対応」
3. `リゼルN型用S内B・キャノン`（ビーム射撃・ヒート式）威力850/2800(集束)、**集束時ユニット貫通効果有**（＝貫通/耐性無視系）
4. `リゼルN型用頭部バルカン砲`（実弾）威力130、弾数50、600発/分、DPS1300、よろけ値6%
5. `リゼルN型用腕部G・ランチャー`（実弾/グレネード）威力1400、2発連続、よろけ値35%x2
6. `リゼルN型用L・B・サーベル`（格闘）威力2600/2800、「空中格闘制御対応」
7. `リゼルN型用S内B・C（変形）`（変形時ビーム射撃）威力1100/2000(集束)
8. `リゼルN型用B・C内蔵シールド`（シールド）※威力表は別系（防御/HP）

判明した重要事実 / パース上の注意:
1. **威力は LV 依存**（LV1/LV2 で異なる）→ 武器は `levels[lv].weapons[]` か weapon内に lv別power を持つ必要（§5案B）。
2. **武器属性の取得（重要・更新）**: ユーザー指摘どおり Wiki では属性が色分けされているが、
   **その色は atwiki の外部CSS由来でスクレイプHTMLにはインライン色が無い**（`<font>`=0、武器名/タブセルの `style` は空、
   ページ内 `color:` 6件はすべて編集注記やステータス表の地色）。→ **色からは判定できない。**
   代わりに **属性はセクション見出しのプレーンテキスト「実弾 / ビーム / 格闘 / シールド / その他」**で構造的に表現されている。
   武装はこのカテゴリ見出しでグルーピングされる → **パーサは「どのカテゴリ節の下にある表か」で属性を確定**するのが正攻法。
   補強として格闘/射撃はカラム構成（クールタイム/武装切替 vs 弾数/リロード/OH）でも判別可、武器名キーワードはフォールバック。
3. **一部の武装表は専用武器ページから transclude されている**（atwiki `#include` / `div.plugin_include`。
   例: ビーム・ライフルは別ページ pageId **7564**）。残りは本体ページ内 `div.box`。
   → パーサは include 展開後のHTML（fetch_page が取得する完成HTML）を見れば両方拾えるが、表のばらつきに注意。
4. **特殊効果は「備考」フリーテキスト**に集中（集束倍率・よろけ値・即撃ち・ユニット貫通・ASL・通常時/変形時 等）。
   → 既存 `extractSkillEffects` 同様の **テキスト解析が必須**。解析不能分は原文保全（[[feedback_data_integrity]]）。
5. テーブルは **rowspan/セル結合**で LV2 行が変化セルのみ（例: LV2 行が威力と射程だけ）→ パーサは結合セルの継承処理が必要。
6. 「通常時/変形時」など **モード違いの同名武器**や「ノン/フル（非集束/集束）」の2値威力がある → スキーマで表現要。

既存のダメージ式（`js/calculator.js` 冒頭コメント）は判明済み:

```
ダメージ = 武器威力 × (100 + 攻撃補正)/100 × (100 − 防御補正)/100 × その他乗算カット
- 攻撃補正(射撃/格闘): 1pt = +1%（上限100 = 与ダメ2倍）
- 防御補正(耐実弾/耐ビーム/耐格闘): 1pt = 1%カット（線形・上限50）
- 三すくみ: 強襲>支援>汎用>強襲、有利+30% / 不利-20%（calculator.js: RAID/SUPPORT/GENERAL 定数）
- 属性ダメージ軽減%（ballistic/beam/melee_damage_cut_pct）は装甲カットと乗算合成
```

つまり「武器威力」という1変数さえ手に入れば基本式は既にある。
問題は **武器ごとの威力・属性・そして §3 の特殊計算**。

### 3-0. deep research 結果（2026-06-10 実施・確定/未確定の総括）

> 出典: gameline.jp 戦闘システム / note.com 弱小賢者（実測検証ブログ）/ atwiki battle-operation2 pages/83.html（3rd Season版・2026-06-05更新）/ 公式 bo2.ggame.jp / fav-reco.work / dengeki。
> 97エージェント・主張25件を3票敵対的検証 → 23確定/2棄却。**公式は式・数値を非公開、確定の大半はコミュニティのリバースエンジニアリング**（パッチで変動しうる）。

**確定（実装してよい）**
- **全体構造＝乗算チェーン（確定/high）**: `ダメージ = 武器威力 ×(機体攻撃補正 × その他攻撃補正) ×(機体防御補正 × その他防御補正) × [格闘方向補正] × [連撃補正] × 属性(三すくみ)補正`。
  補正適用順は **攻撃力 → 防御力 → 格闘方向補正 → 連撃補正 → 三すくみ**。基本式 `威力×(100+攻撃補正)/100×(100−防御補正)/100` は実測一致（2875×139/100×74/100=2957）。**既存 calculator.js の式が正しいと裏付けられた**。
- **攻撃補正（確定/high）**: 1pt=+1%、`(100+補正)/100`、上限100（=+100%=×2でカンスト。100超は計算されない）。
- **防御補正/属性耐性（確定/high）**: 1pt=−1% の**線形**カット、`(100−補正)/100`、**逓減なし**、属性耐性の実効上限50（=50%カット）。←旧逓減近似が誤りで線形が正しいことを再確認。
- **端数処理（確定/high・新発見・実装注意）**: 各補正計算は**小数第3位以下を切り捨て**（=小数2桁）、武器威力との乗算時は**その都度 floor で整数化**。round-to-nearest ではなく **floor**。
- **三すくみ（確定/high・公式裏付け）**: 循環 **汎用>強襲>支援>汎用**（既存 calculator.js の TYPE_ADVANTAGE 方向と一致）。有利側は**与ダメ増・被ダメ減の双方向**。式末尾に `×(1+属性補正値/100)` として乗る。⚠**正確な%は諸説あり/パッチ変動**（よく挙がる値は有利+30%/不利−20%＝既存実装の 1.30/0.80 と整合）。
- **スキル合成順（確定/medium・Q9）**: **異カテゴリのバフ/デバフは乗算**で重なる。**同カテゴリ（同種補正値・格闘補正プログラム等）は加算**で1値にまとまる。オーバーチューン等の追加補正項は基本ダメージにさらに `×(100+補正)/100` で独立乗算。スコープ/高精度スナイプ補正は独立層でなく攻撃補正項（上限100）に畳み込まれる。
- **蓄積よろけ（確定/high・Q10一部）**: 隠しパラメータ。武器別固定値の合計が **>100 でよろけ**（例 100mmMG=7・バズーカ=80・バルカン=2・ショットガン=35/ペレット）。最終命中から **3秒でハードリセット**（漸減なし）。リアクション中は蓄積しない。ダメージコントロール=よろけ閾値を引上げ（Lv1 130%/Lv2 160%/Lv3 200%）、マニューバーアーマーLv2=被蓄積 ×0.8（floor）。

**未確定（実装は暫定値・要追加調査）**
- **Q2 特殊属性の帰属（未確定）**: 実弾→耐実弾/ビーム→耐ビーム/格闘→耐格闘 の基本対応は前提化されているが、榴弾/グレネード/ミサイル/爆発/火炎が**どの防御補正か**を明示する確定出典なし。→ 暫定: 実弾扱い。
- **Q3 耐性無視/ユニット貫通（未確定）**: 割合無視か固定無視か・どの段で効くか、確定出典なし。→ 暫定: ×1（無効化なし）。
- **Q4 格闘方向補正・連撃補正（部分確定）**: 式の項 `[格闘方向補正]` `[連撃補正]` の**存在は確定**したが、正面/側面/背面の**正確な倍率**・コンボ段数別威力・ヘビーアタック/よろけ追撃倍率は**数値が確定できず**。→ 暫定: front/side/back すべて1.0、単発のみ。
- **Q6 よろけ/ダウン/空中への与ダメUP・ダウン追撃（未確定）**: 確定出典なし。→ ▶後段（等倍）。
- **Q7 部位/急所別（未確定）**: atwiki 自身が「局部ダメージのみ再現不可」と明言。具体値不明。→ ▶後段。
- **Q8 距離減衰（未確定）** / **Q10 集束・変形の威力モード（未確定・ただしデータ駆動）**: 確定出典なし。集束/変形は威力が武装表に別値で載るため**式でなくデータで扱う**（§5-2 modes）。

  "hits": 1,                       // 連射/多段ヒット数

## 6. 計算エンジン設計（calculator.js 拡張・提案）

新規ピュア関数（テスト容易・既存スタイル踏襲）:

```
calcWeaponDamage(weapon, attacker, defender, opts) -> {
  perHit, totalPerVolley, dps?, breakdown:{ basePower, atkCorrMul, defCutMul,
    triadMul, resistIgnore, directionMul, ... }
}
```

- `attacker` / `defender` は既存 `applyParts` の結果（modified ステータス）を流用。
  → **保存構成2つから `calcStatsFromBuild` で算出した modified を渡せば再利用できる**。
- 防御補正カットは既存 `calcCutRate`（線形・上限50）＋属性ダメージ軽減%の乗算合成を流用。
- 三すくみは既存 RAID/SUPPORT/GENERAL 定数を流用。
- **耐性無視 / 格闘方向補正** は本関数で新規追加（§3-2の確定式に従う）。
- `breakdown` を返して計算方法モーダルやツールチップで内訳を可視化（[[feedback_terminology]]: 用語正確に）。

> 既存の `calcOffenseScore` は「武器なし抽象スコア」のまま温存し、本機能とは別経路にする
> （回帰リスク隔離）。

## 9. リスク・未確定事項

- ~~データ取得の可否が最大リスク~~ → §2-2 で **武装データの存在を確認済み**。リスクは
  「**武装表パーサの頑健性**」へ移行。武器種別でカラム構成が異なり（射撃/格闘）、rowspan結合・
  LV別威力・モード違い同名武器・備考フリーテキストがあるため **スキーマ揺れへの耐性**が要点。
  Phase1で複数機体（変形機/支援機/格闘機）を実データ検証してパーサの網羅性を確かめる。
- 特殊効果（耐性無視・方向補正・条件付きバフ）は機体/武器ごとに表記揺れが大きい想定。
  解析できない分は `notes` 原文保全＋UIで「未モデル」明示（過大評価を避ける）。
- スコープ膨張注意。まずは **「1発/斉射の素ダメージ＋属性カット＋三すくみ＋方向補正＋耐性無視」**
  の静的計算に絞る。DPS・距離・部位は後段（[[feedback_no_half_finished]] と矛盾しないよう、
  Phase範囲を「静的ダメージ計算」として1機能完結に切る）。
